# Agent 核心架构设计 — 系统化学习大纲

> Agent 架构师的核心工作是设计 **"循环（Loop）"** 和 **"状态（State）"**，而不仅仅是调 API。
> 建议学习周期：8-12 周，每周 6-8 小时。

---

## 一、Agent 基础概念

### 1.1 什么是 Agent

- **定义**：Agent = LLM + Planning + Memory + Tools
- **一句话理解**：Agent 是一个能**自主感知环境、制定计划、调用工具、完成目标**的智能系统，LLM 是它的大脑，但不是它的全部

#### Agent vs Chatbot 的本质区别

| 维度 | Chatbot | Agent |
|------|---------|-------|
| 交互模式 | 单轮问答，用户驱动每一步 | 自主循环，接受目标后自行规划执行 |
| 状态管理 | 无状态或简单上下文窗口 | 有持久化状态，支持中断恢复 |
| 工具使用 | 不调用外部工具 | 主动调用搜索、数据库、API 等 |
| 错误处理 | 返回错误信息给用户 | 自主检测错误、反思原因、重试修复 |
| 目标驱动 | 响应式（你问我答） | 目标式（给定目标，自主完成） |

**举例**：用户说"帮我调研竞品 X 的最新动态并写一份分析报告"
- **Chatbot**：基于已有知识生成一段文本，可能过时且无法验证
- **Agent**：拆分任务 → 搜索竞品新闻 → 抓取官网信息 → 整合分析 → 生成报告 → 自检质量

#### Agent 的核心能力模型

```
                    ┌──────────────┐
                    │   用户目标    │
                    └──────┬───────┘
                           ▼
                 ┌─────────────────────┐
                 │   感知 Perception    │  理解用户意图、解析上下文
                 └─────────┬───────────┘
                           ▼
                 ┌─────────────────────┐
                 │   规划 Planning      │  拆解任务、制定执行步骤
                 └─────────┬───────────┘
                           ▼
                 ┌─────────────────────┐
          ┌─────▶│   行动 Action        │  调用工具、执行操作
          │      └─────────┬───────────┘
          │                ▼
          │      ┌─────────────────────┐
          └──────│   反思 Reflection    │  评估结果、自我修正
                 └─────────────────────┘
                           ▼
                    ┌──────────────┐
                    │   任务完成    │
                    └──────────────┘
```

- **感知（Perception）**：理解用户输入的真实意图，感知当前任务进度与环境变化
- **规划（Planning）**：将复杂目标拆解为可执行的子任务序列，决定执行顺序与依赖关系
- **行动（Action）**：选择并调用合适的工具（搜索、代码执行、API 调用），执行具体操作
- **反思（Reflection）**：检查行动结果是否符合预期，识别错误并调整策略

#### Agent 的演进脉络

```
Chatbot（纯对话）
  → CoT（思维链，让模型先思考再回答）
    → ReAct（思考+行动交替循环）
      → Plan-and-Execute（先规划再执行）
        → Multi-Agent（多智能体协作）
          → Agentic Workflow（DAG 编排，生产级工作流）
```

### 1.2 Agent 的核心组成

| 组件 | 作用 | 类比 | 关键技术 |
|------|------|------|----------|
| LLM | 大脑，负责推理和决策 | CPU | GPT-4、Claude、Qwen |
| Memory | 记忆系统，维护上下文 | 内存 + 硬盘 | 向量库、会话摘要、实体记忆 |
| Tools | 外部能力扩展 | 外设接口 | Function Calling、MCP |
| Planning | 任务分解与编排 | 操作系统调度器 | CoT、ReAct、LangGraph |

#### 1.2.1 LLM — 推理引擎

LLM 是 Agent 的核心大脑，决定了 Agent 的推理上限：

- **能力边界**：
  - 自然语言理解与生成
  - 逻辑推理与常识判断
  - 指令遵循与格式化输出（JSON / 代码 / 结构化文本）
  - **不能**：精确数学计算、实时信息获取、确定性逻辑（需要工具弥补）
- **选型考量**：
  - 推理能力（复杂任务用 GPT-4 / Claude Opus，简单任务用 Haiku / GPT-4o-mini 降本）
  - 上下文窗口大小（影响单次可处理的信息量）
  - Function Calling 支持质量（直接影响工具调用准确率）
  - 延迟与成本（生产环境的核心约束）

#### 1.2.2 Memory — 记忆系统

解决"聊着聊着就忘了"和"上下文窗口爆炸"的问题：

```
┌─────────────────────────────────────────────────────┐
│                    Memory 架构                       │
│                                                      │
│  ┌──────────────────┐    ┌───────────────────────┐   │
│  │   短期记忆         │    │   长期记忆             │   │
│  │  (Working Memory)  │    │  (Long-term Memory)    │   │
│  ├──────────────────┤    ├───────────────────────┤   │
│  │ • 当前对话上下文   │    │ • 向量库存储历史交互   │   │
│  │ • 滑动窗口         │    │ • 实体记忆（用户偏好） │   │
│  │ • Token 预算管理   │    │ • 摘要压缩归档         │   │
│  │ • 本轮工具调用记录 │    │ • 知识图谱关联         │   │
│  └──────────────────┘    └───────────────────────┘   │
│                                                      │
│  ┌──────────────────────────────────────────────┐    │
│  │   上下文工程 (Context Engineering)             │    │
│  ├──────────────────────────────────────────────┤    │
│  │ • 信息压缩：摘要替代全量历史                    │    │
│  │ • 注意力管理：过滤噪音，保留关键信息             │    │
│  │ • 检索注入：按需从长期记忆中提取相关片段          │    │
│  └──────────────────────────────────────────────┘    │
└─────────────────────────────────────────────────────┘
```

- **短期记忆**：当前会话的上下文，通常直接放在 Prompt 中
  - **滑动窗口（Sliding Window）**：仅保留最近 N 轮对话
  - **Token 预算管理**：动态计算 Token 消耗，超限时触发压缩
- **长期记忆**：跨会话持久化存储
  - **向量存储**：用 Chroma / Milvus / Pinecone 存储历史交互的 Embedding
  - **实体记忆**：提取对话中的实体（用户姓名、偏好、提及的项目），存入结构化存储
  - **记忆检索**：每轮对话前，根据用户 Query 检索相关历史，注入到 System Prompt

#### 1.2.3 Tools — 工具系统

工具是 Agent 与真实世界交互的桥梁，弥补 LLM 的能力短板：

| 工具类型 | 示例 | 弥补的能力短板 |
|----------|------|----------------|
| 信息检索 | 搜索引擎、知识库查询 | 实时信息获取 |
| 数据操作 | 数据库 CRUD、文件读写 | 持久化数据操作 |
| 代码执行 | Python REPL、Shell | 精确计算、复杂逻辑 |
| 外部 API | 发邮件、Slack 消息、GitHub 操作 | 与外部系统交互 |
| 多模态 | 图片识别、语音转文字 | 非文本输入处理 |

- **工具设计原则**：
  - 每个工具职责单一，做一件事并做好
  - 工具描述（description）要对模型友好，清晰说明何时使用
  - 参数定义严格（类型、必填、枚举），减少模型自由发挥空间
  - 返回值结构化，便于模型解析
- **Skill 封装**：将原子工具组合成高阶能力
  - 原子工具：`search_google`、`read_webpage`、`summarize_text`
  - 高阶 Skill：`research_competitor`（搜索 → 抓取 → 摘要 → 分析）

#### 1.2.4 Planning — 规划系统

让 LLM 学会"三思而后行"，从"一步到位"到"分步执行"：

| 规划策略 | 描述 | 适用场景 |
|----------|------|----------|
| 无规划 | 直接生成答案 | 简单问答 |
| CoT（思维链） | 生成前先推理 | 需要逻辑推导的问题 |
| ReAct | 思考-行动交替循环 | 需要工具辅助的任务 |
| Plan-and-Execute | 先生成完整计划，再逐步执行 | 多步骤复杂任务 |
| DAG 编排 | 构建有向无环图，支持并行和分支 | 生产级工作流 |

### 1.3 理解 Agent 的关键思维转变

从"调 API"到"设计 Agent"，需要以下思维转变：

| 旧思维 | 新思维 |
|--------|--------|
| 一次 API 调用解决问题 | 多次循环迭代逼近目标 |
| 输出就是最终结果 | 输出是下一步行动的输入 |
| 错误返回给用户处理 | Agent 自主检测错误并修复 |
| 模型什么都要做 | 模型只做推理，计算交给工具 |
| 固定流程，线性执行 | 动态决策，条件分支，循环重试 |
| 无状态的请求-响应 | 有状态的持久化执行 |

## 二、Agent 运行模式：从 Chat 到 Workflow

> 本章按复杂度递进，从最简单的单步调用到生产级 DAG 编排。

### 2.1 单 Agent 模式 `🟢 入门`

#### 2.1.1 ReAct 循环（Reasoning + Acting）

**核心思想**：让模型交替执行「思考」和「行动」，形成 Thought → Action → Observation 的闭环循环。

**执行流程图**：

```
用户输入任务
    │
    ▼
┌──────────────────────────────────────┐
│           ReAct 循环                  │
│                                      │
│   ┌──────────┐                       │
│   │ Thought  │ ← 分析当前状态        │
│   │ (思考)   │   决定下一步行动       │
│   └────┬─────┘                       │
│        ▼                             │
│   ┌──────────┐                       │
│   │ Action   │ ← 选择工具并调用      │
│   │ (行动)   │   或生成最终答案       │
│   └────┬─────┘                       │
│        ▼                             │
│   ┌──────────┐     ┌──────────┐      │
│   │Observation│────▶│ 是否完成？│      │
│   │ (观察)   │     └────┬─────┘      │
│   └──────────┘       否 │ 是         │
│        ▲               │  │          │
│        └───────────────┘  │          │
│                           ▼          │
│                     最终输出          │
└──────────────────────────────────────┘
```

**ReAct Prompt 模板示例**：

```
你是一个有用的 AI 助手，可以使用以下工具：
{tools_description}

请严格按照以下格式回答：

Thought: 我需要思考如何解决这个问题...
Action: tool_name
Action Input: {"param": "value"}
Observation: [工具返回的结果]
... (重复 Thought/Action/Observation)
Thought: 我现在知道最终答案了
Final Answer: 最终答案
```

**ReAct 执行实例**（以"北京今天天气怎么样？"为例）：

```
Thought: 用户问的是北京今天的天气，我没有实时信息，需要调用搜索工具。
Action: search
Action Input: {"query": "北京今天天气"}
Observation: 北京今日天气：晴，气温 15-25°C，西北风 3-4 级...

Thought: 我已经获取到天气信息，可以直接回答用户。
Final Answer: 北京今天天气晴朗，气温 15-25°C，有西北风 3-4 级，适合外出。
```

**学习要点**：
- **Prompt 设计**：格式化输出是 ReAct 的基础，模型必须严格按照 Thought/Action/Observation 格式输出
- **终止条件**：必须明确设定最大循环次数（通常 5-10 次），防止无限循环
- **错误处理**：工具调用失败时，让模型在下一轮 Thought 中分析原因并调整策略
- **工具选择**：当有多个工具可用时，模型需要在 Thought 中推理选择最合适的

**常见陷阱**：

| 陷阱 | 现象 | 解决方案 |
|------|------|----------|
| 无限循环 | Agent 反复调用同一工具 | 设置 max_iterations + 去重检测 |
| 格式漂移 | 输出渐渐偏离约定格式 | 在每轮 Prompt 中重申格式要求 |
| 工具幻觉 | 调用不存在的工具 | 严格限制可用工具列表 |
| 观察忽视 | 忽略工具返回的结果 | 强制在 Thought 中引用 Observation |

#### 2.1.2 Plan-and-Execute 模式

**核心思想**：先让模型生成完整的执行计划，再逐步执行每个步骤。与 ReAct 的区别是「先全局规划，后逐步执行」。

**执行流程**：

```
用户输入任务
    │
    ▼
┌────────────────┐
│   Planner       │  生成完整的步骤计划
│   (规划器)      │  [Step1, Step2, Step3, ...]
└───────┬────────┘
        ▼
┌────────────────┐
│   Executor      │  逐步执行计划中的每个步骤
│   (执行器)      │  每步可调用工具
└───────┬────────┘
        ▼
┌────────────────┐
│   Re-planner    │  根据执行结果判断：
│   (重规划器)    │  ✅ 任务完成 → 输出结果
└────────────────┘  ❌ 需要调整 → 更新计划
```

**与 ReAct 的对比**：

| 维度 | ReAct | Plan-and-Execute |
|------|-------|------------------|
| 规划时机 | 每一步实时决策 | 先生成完整计划 |
| 全局视野 | 较弱，容易局部最优 | 较强，有全局规划 |
| 灵活性 | 高，随时调整 | 中等，需要 re-plan |
| 适用场景 | 探索性任务、步骤不确定 | 步骤较明确的多步任务 |
| 复杂度 | 低 | 中 |

**实例**（以"帮我调研 LangGraph 框架并写一份技术评估报告"为例）：

```
=== Planner 输出 ===
计划：
1. 搜索 LangGraph 的官方文档和最新版本信息
2. 搜索 LangGraph 的社区评价和使用案例
3. 与其他框架（LangChain、AutoGen）进行对比
4. 整合信息，撰写技术评估报告

=== Executor 逐步执行 ===
执行 Step 1: search("LangGraph official documentation 2025")
执行 Step 2: search("LangGraph reviews use cases")
执行 Step 3: search("LangGraph vs LangChain vs AutoGen comparison")
执行 Step 4: 基于以上信息生成报告

=== Re-planner 判断 ===
✅ 所有步骤已完成，输出最终报告
```

#### 2.1.3 自主反思（Self-Reflection）

**核心思想**：让 Agent 对自己的输出进行评估，从失败中学习改进。

**Reflexion 框架**：

```
         ┌─────────────────────────────────────┐
         │          Reflexion 循环               │
         │                                      │
  任务 ──▶  执行 ──▶ 评估 ──▶ 反思 ──▶ 改进    │
         │   │        │       │        │        │
         │   │        ▼       │        ▼        │
         │   │    成功？───是──┼──▶ 输出结果     │
         │   │        │       │                 │
         │   │       否       │                 │
         │   │        ▼       ▼                 │
         │   │    生成反思笔记                   │
         │   │    "上次失败因为X，               │
         │   │     下次应该Y"                   │
         │   │        │                         │
         │   └────────┘ (带着反思重新执行)       │
         └─────────────────────────────────────┘
```

**反思的三个层次**：

| 层次 | 描述 | 示例 |
|------|------|------|
| 结果反思 | 检查输出是否满足要求 | "答案缺少数据来源，需要补充引用" |
| 过程反思 | 回顾执行路径是否最优 | "应该先查官方文档，而不是直接搜索博客" |
| 策略反思 | 总结可复用的经验教训 | "类似的调研任务，优先从官方渠道获取一手信息" |

**自修复逻辑 — 代码执行场景**：

```
第1次执行：
  Code: result = 10 / 0
  Error: ZeroDivisionError: division by zero

反思：代码中存在除零错误，需要添加零值检查

第2次执行（自动修复）：
  Code: result = 10 / x if x != 0 else "除数不能为零"
  Result: 成功 ✅
```

**自修复逻辑 — 工具调用场景**：

```
第1次尝试：
  Action: query_database(sql="SELECT * FROM users WHERE name = 张三")
  Error: SQL 语法错误，字符串需要引号

反思：SQL 字符串值需要用单引号包裹

第2次尝试（自动修复）：
  Action: query_database(sql="SELECT * FROM users WHERE name = '张三'")
  Result: 成功 ✅
```

#### 2.1.4 单 Agent 模式小结

```
选择指南：
┌─────────────────────────────────────────────────┐
│ 任务类型                →  推荐模式              │
├─────────────────────────────────────────────────┤
│ 步骤不确定，需要探索     →  ReAct                │
│ 步骤较明确，多步任务     →  Plan-and-Execute      │
│ 需要质量保证，容忍重试   →  加 Self-Reflection    │
│ 以上组合               →  ReAct + Reflection     │
└─────────────────────────────────────────────────┘
```

### 2.2 状态机设计 `🟡 进阶`

#### 2.2.1 为什么需要状态机？

**问题引入**：简单的 ReAct 循环在以下场景会遇到困难：
- 对话中断后无法恢复（无持久化状态）
- 流程中需要人工审批节点
- 不同阶段需要不同的处理逻辑（如：收集信息 → 确认 → 执行 → 汇报）
- 需要根据条件走不同的分支

**状态机的价值**：将 Agent 的行为建模为**确定的状态转移**，每个状态有明确的入口、出口和行为。

#### 2.2.2 有限状态机（FSM）建模

**核心概念**：Agent 在任意时刻处于一个明确的状态，根据输入/事件转移到下一个状态。

**关键要素**：
- **State（状态）**：Agent 当前所处的阶段
- **Transition（转移）**：从一个状态到另一个状态的条件
- **Action（动作）**：进入/离开某状态时执行的操作
- **Guard（守卫）**：转移前的条件检查

**示例 — 客服 Agent 状态机**：

```
┌──────────┐   用户输入    ┌───────────┐  已分类    ┌──────────┐
│          │──────────────▶│           │──────────▶│          │
│  待接入   │              │  意图识别  │           │  处理中   │
│          │◀──────────────│           │  未识别   │          │
└──────────┘   重新询问    └───────────┘           └────┬─────┘
                                                       │
                           ┌───────────┐               │ 处理完成
                           │           │◀──────────────┘
                           │  确认满意  │
                           │           │──── 不满意 ──▶ 转人工
                           └─────┬─────┘
                                 │ 满意
                                 ▼
                           ┌───────────┐
                           │  对话结束  │
                           └───────────┘
```

**状态定义示例**：

```python
# 伪代码：客服 Agent 状态定义
states = {
    "idle":         "等待用户输入",
    "classifying":  "识别用户意图",

    "confirming":   "确认用户是否满意",

transitions = [
```

#### 2.2.3 LangGraph 状态图

**LangGraph 是什么**：基于图（Graph）的 Agent 编排框架，解决传统 LangChain Chain 难以处理的**循环**和**条件分支**问题。

**三大核心概念**：

| 概念 | 作用 | 类比 |
|------|------|------|
| State | 共享的状态对象，贯穿整个执行流程 | 全局变量 |
| Node | 处理节点，接收 State 并返回更新 | 函数 |
| Edge | 节点之间的连接，支持条件跳转 | if-else 分支 |

**LangGraph 代码结构**（伪代码）：

```python
from langgraph.graph import StateGraph, END

# 1. 定义状态结构
class AgentState(TypedDict):
    messages: list          # 对话历史
    current_step: str       # 当前步骤
    tool_results: list      # 工具调用结果
    retry_count: int        # 重试次数

# 2. 定义节点（处理函数）
def classify_intent(state: AgentState) -> AgentState:
    """意图识别节点"""
    # 调用 LLM 分类用户意图
    ...

def call_tool(state: AgentState) -> AgentState:
    """工具调用节点"""
    # 根据意图选择并调用工具
    ...

def check_result(state: AgentState) -> AgentState:
    """结果检查节点"""
    # 评估工具调用结果是否满足要求
    ...

# 3. 定义路由（条件边）
def should_retry(state: AgentState) -> str:
    if state["retry_count"] > 3:
        return "give_up"       # 超过重试上限
    if is_success(state):
        return "success"       # 成功
    return "retry"             # 重试

# 4. 构建图
graph = StateGraph(AgentState)
graph.add_node("classify", classify_intent)
graph.add_node("tool_call", call_tool)
graph.add_node("check", check_result)

graph.add_edge("classify", "tool_call")
graph.add_edge("tool_call", "check")
graph.add_conditional_edges("check", should_retry, {
    "success": END,
    "retry":   "tool_call",    # 循环！
    "give_up": END,
})

# 5. 编译并运行
app = graph.compile()
result = app.invoke({"messages": [user_input]})
```

**LangGraph 的核心优势**：

```
LangChain (Chain)          LangGraph (Graph)
A → B → C → D             A → B ──┐
                                   ▼
（只能顺序执行）            C ← 条件判断 → D
                                   │
                           （支持循环、分支、并行）
```

| 能力 | LangChain Chain | LangGraph |
|------|-----------------|-----------|
| 顺序执行 | ✅ | ✅ |
| 条件分支 | ❌ 困难 | ✅ conditional_edges |
| 循环重试 | ❌ 不支持 | ✅ 边可回指 |
| 状态持久化 | ❌ | ✅ Checkpoint |
| 中断恢复 | ❌ | ✅ thread_id |
| 人工确认节点 | ❌ | ✅ interrupt_before |

**Checkpoint 持久化机制**：

```
对话 Thread #abc123
    │
    ├── Checkpoint 1: [意图识别完成]      ← 可回溯
    ├── Checkpoint 2: [工具调用完成]      ← 可回溯
    ├── Checkpoint 3: [等待人工确认] ★    ← 中断点
    │
    ... (用户关闭浏览器)
    ... (第二天重新打开)
    │
    └── 从 Checkpoint 3 恢复 ★            ← 断点续跑
        └── Checkpoint 4: [任务完成]
```

- 每个 Node 执行后自动保存 Checkpoint
- 通过 `thread_id` 标识对话，支持跨会话恢复
- 支持回溯到任意历史 Checkpoint 重放

**Human-in-the-loop 设计**：

```python
# 在关键节点前中断，等待人工确认
graph.compile(
    checkpointer=memory_saver,
    interrupt_before=["execute_dangerous_action"]  # 在此节点前暂停
)

# 人工确认后继续执行
app.invoke(None, config={"configurable": {"thread_id": "abc123"}})
```

适用场景：
- 高危操作前确认（删除数据、发送邮件、支付）
- 审批流程（生成的报告需要主管审核）
- 纠偏介入（Agent 走错方向时人工修正）

#### 2.2.4 SOP 硬编码模式

**核心思想**：将业务的标准操作流程（SOP）硬编码为 Agent 的状态转移约束。

**适用场景**：
- 客服 SOP：问候 → 收集信息 → 查询系统 → 提供方案 → 确认满意度
- 销售 SOP：需求挖掘 → 产品推荐 → 异议处理 → 促成成交
- 审核 SOP：接收材料 → 格式检查 → 内容审核 → 出具结论

**与自由 Agent 的区别**：

| 维度 | 自由 Agent | SOP Agent |
|------|-----------|-----------|
| 行为确定性 | 低，LLM 自主决策 | 高，按预定流程执行 |
| 可控性 | 低 | 高 |
| 灵活性 | 高 | 低 |
| 适用场景 | 探索性任务 | 标准化业务流程 |

#### 验收标准
- 能用 LangGraph 实现一个带条件分支和循环重试的 Agent
- 工具调用失败可自动重试，不因单点失败导致整链路退出
- 实现 Checkpoint 持久化，支持中断后恢复（通过 thread_id）
- 状态快照可回放最近 5 步执行路径

### 2.3 Agentic Workflow `🔴 专家`

#### 2.3.1 从 Agent 到 Workflow 的演进

```
单 Agent (ReAct)         多步 Agent (状态机)         Agentic Workflow (DAG)

  循环执行                 状态转移                    任务编排
  灵活但不可控             可控但较简单                 可控且可扩展

  适合探索性任务           适合固定流程                 适合生产级复杂任务
```

**为什么需要 Workflow？**
- 复杂任务有明确的子任务依赖关系
- 需要并行执行独立子任务以提升效率
- 需要精细的错误处理（某个节点失败不影响其他节点）
- 需要可观测性（每个节点的输入输出、耗时、状态）

#### 2.3.2 DAG（有向无环图）编排

**DAG 的设计思路**：将复杂任务拆解为一组节点和边组成的执行图，节点是具体操作，边定义执行顺序和条件。

**节点类型详解**：

| 节点类型 | 职责 | 示例 |
|----------|------|------|
| **LLM 节点** | 需要模型推理 | 意图分类、内容生成、摘要提取 |
| **工具节点** | 调用外部 API / 数据库 | 搜索、查询数据库、发消息 |
| **路由节点** | 根据条件决定分支 | 根据意图分流到不同处理链 |
| **人工节点** | Human-in-the-loop 确认 | 审批、纠偏、质检 |
| **聚合节点** | 合并多个并行分支结果 | 多源搜索结果汇总 |
| **转换节点** | 数据格式转换 | JSON → Markdown、结构化提取 |

**实际案例 — 智能报告生成 Workflow**：

```
            ┌─────────────────────────────────────────┐
            │           用户输入：生成竞品分析报告       │
            └───────────────────┬─────────────────────┘
                                │
                    ┌───────────▼───────────┐
                    │   任务理解 (LLM 节点)   │
                    │   提取：竞品名称、维度    │
                    └───────────┬───────────┘
                                │
              ┌─────────────────┼─────────────────┐
              ▼                 ▼                 ▼
    ┌──────────────┐  ┌──────────────┐  ┌──────────────┐
    │ 搜索官网信息  │  │ 搜索媒体报道  │  │ 搜索用户评价  │
    │ (工具节点)   │  │ (工具节点)   │  │ (工具节点)   │
    └──────┬───────┘  └──────┬───────┘  └──────┬───────┘
           │                 │                 │
           └─────────────────┼─────────────────┘
                             ▼
                   ┌──────────────────┐
                   │  信息聚合 (聚合)  │
                   └────────┬─────────┘
                            ▼
                   ┌──────────────────┐
                   │  报告生成 (LLM)   │
                   └────────┬─────────┘
                            ▼

                   └────────┬─────────┘
                       通过  │  不通过
                       ┌────┘└────┐
                       ▼          ▼
                  输出报告    反馈修改意见
                              → 回到报告生成
```

#### 2.3.3 容错机制设计

**节点级容错**：

```python
# 伪代码：带重试的节点执行
class NodeExecutor:
    def execute_with_retry(self, node, state, max_retries=3):
        for attempt in range(max_retries):
            try:
                result = node.run(state)
                return result
            except ToolError as e:
                if attempt < max_retries - 1:
                    wait_time = 2 ** attempt  # 指数退避
                    sleep(wait_time)
                    state["error_context"] = str(e)  # 传递错误上下文
                else:
                    return self.fallback(node, state)  # 降级方案
```

**三层容错体系**：

| 层次 | 机制 | 说明 |
|------|------|------|
| **节点级** | 重试 + 指数退避 | 单个节点失败后自动重试，等待时间逐次增加 |
| **子图级** | 子图回滚 | 一组相关节点失败后，回滚到子图入口状态 |
| **全局级** | 断点续跑 | 从最近成功的 Checkpoint 恢复，跳过已完成节点 |

**降级策略示例**：

```
主路径：调用 GPT-4 生成报告
  │
  ├── 失败（超时/限流）
  │     ▼
  │   降级1：切换到 Claude 重试
  │     │
  │     ├── 失败
  │     │     ▼
  │     │   降级2：使用缓存的模板 + 填充数据
  │     │     │
  │     │     ├── 失败
  │     │     │     ▼
  │     │     │   降级3：返回原始数据 + 人工处理提示
```

#### 2.3.4 编排模式详解

| 模式 | 流程 | 适用场景 | 示例 |
|------|------|----------|------|
| **Sequential** | A → B → C | 步骤间有严格依赖 | 数据清洗 → 分析 → 报告 |
| **Parallel** | A ┬ B → D | 独立子任务可并行 | 同时搜索多个数据源 |
|              |   └ C ┘   |                  |                       |
| **Router** | A → 条件 → B/C/D | 根据输入走不同分支 | 意图分类后路由到不同处理器 |
| **Map-Reduce** | [A1,A2,...] → 聚合 → B | 大量同类子任务 | 多文档摘要 → 合并总结 |
| **Pipeline** | A → B → 人工 → C | 含人工审核的流水线 | 内容生成 → 审核 → 发布 |

**Map-Reduce 实例 — 多文档摘要**：

```
输入：10 篇论文 PDF

Map 阶段（并行）：
  ├── 论文1 → LLM 摘要 → 摘要1
  ├── 论文2 → LLM 摘要 → 摘要2
  ├── ...
  └── 论文10 → LLM 摘要 → 摘要10

Reduce 阶段：
  [摘要1, 摘要2, ..., 摘要10] → LLM 综合分析 → 最终综述报告
```

#### 2.3.5 设计原则

1. **最小权限**：每个节点只获取它需要的数据和工具权限
2. **幂等性**：节点可安全重复执行，不会产生副作用（便于重试）
3. **可观测性**：每个节点记录输入、输出、耗时、状态，便于调试和监控
4. **渐进增强**：先实现最简版本（Sequential），再逐步添加并行、容错、人工节点

#### 验收标准
- 能设计一个包含并行节点、路由节点和人工确认节点的 DAG Workflow
- 实现节点级重试 + 指数退避 + 降级方案
- 支持断点续跑：中断后可从最近成功 Checkpoint 恢复
- 有完整的节点级 Tracing（输入/输出/耗时/状态）

### 2.4 运行模式总结与选型

```
┌─────────────────────────────────────────────────────────┐
│                   Agent 运行模式选型指南                   │
├───────────────┬──────────┬──────────┬───────────────────┤
│     模式       │  复杂度  │  可控性  │  适用场景          │
├───────────────┼──────────┼──────────┼───────────────────┤
│ ReAct          │  ⭐      │   低    │ 探索性任务、原型    │
│ Plan-Execute   │  ⭐⭐    │   中    │ 多步骤明确任务     │
│ 状态机 (FSM)   │  ⭐⭐⭐  │   高    │ 固定业务流程       │
│ LangGraph      │  ⭐⭐⭐  │   高    │ 需要循环/分支/持久化│
│ DAG Workflow   │  ⭐⭐⭐⭐ │  很高   │ 生产级复杂编排     │
└───────────────┴──────────┴──────────┴───────────────────┘

选择建议：
• 先跑通 ReAct，理解 Agent 基本循环
• 需要可控性 → 引入状态机 / LangGraph
• 需要生产化 → 上 DAG Workflow + 容错 + 监控
• 不要过度设计：简单任务用 ReAct 就够了
```

### 3.1 Function Calling 深度实践 `🟢 入门`

### 3.2 MCP（Model Context Protocol） `🟡 进阶`

#### 协议概述
- **定位**：Agent 连接外部数据和工具的标准化协议，类似 USB 接口
- **解决的问题**：每个数据源都要写独立集成代码 → 统一接口标准
- **架构**：Client-Server 模式
  - **MCP Server**：暴露工具（Tools）、资源（Resources）、提示模板（Prompts）
  - **MCP Client**：Agent 侧的协议客户端，发现并调用 Server 能力
  - **传输层**：支持 stdio、HTTP SSE 等通信方式

#### 核心能力

| 能力 | 描述 | 示例 |
|------|------|------|
| Tools | 可调用的函数 | 查询数据库、发送消息 |
| Resources | 可读取的数据 | 文件内容、API 响应 |
| Prompts | 预定义的提示模板 | 代码审查模板、摘要模板 |

#### 实战路线
1. 搭建一个 MCP Server 连接本地 SQLite / 文件系统
2. 用 Claude Desktop 或自定义 Client 接入
3. 实现多 Server 组合（文件 + 数据库 + 搜索引擎）

### 3.3 A2A（Agent-to-Agent Protocol） `🔴 专家`

#### 协议概述
- **提出者**：Google（2024）
- **目标**：解决不同平台、不同框架构建的 Agent 之间的互操作问题
- **核心概念**：
  - **Agent Card**：Agent 的能力名片，描述其技能和接口
  - **Task**：Agent 间交互的基本单元
  - **Message / Artifact**：任务中的通信内容和产物

#### 与 MCP 的关系

| 维度 | MCP | A2A |
|------|-----|-----|
| 连接对象 | Agent ↔ 工具/数据 | Agent ↔ Agent |
| 类比 | USB 接口 | 网络协议 |
| 关系 | 互补，非竞争 | 互补，非竞争 |

## 四、Agent 安全体系

### 4.1 Prompt Injection 防御 `🟢 入门`

- **攻击类型**：
  - **直接注入**：用户在输入中嵌入恶意指令
  - **间接注入**：通过工具返回的数据注入恶意指令（如网页内容、邮件正文）
- **防御策略**：
  - 输入清洗与敏感指令检测
  - System Prompt 与 User Input 的隔离
  - 沙箱执行环境：限制 Agent 可访问的资源范围
  - 关键操作前的确认机制（Human-in-the-loop）

### 4.2 Tool Abuse 防护 `🟡 进阶`

- **风险场景**：模型被诱导反复调用高危工具（如删除文件、发送邮件）
- **防护机制**：
  - 工具调用频率限制（Rate Limiting）
  - 工具权限分级（只读 / 读写 / 危险操作）
  - 调用链审计日志
  - 高危操作的二次确认

### 4.3 数据泄露防护 `🔴 专家`

- **敏感信息过滤**：PII（个人可识别信息）检测与脱敏
- **输出审计**：Agent 响应的合规性检查
- **系统信息隔离**：防止 Agent 在对话中泄露 System Prompt、内部 API Key 等

- Human-in-the-loop 节点设计
- 子图（Subgraph）组合

### 6.2 AutoGen `🟡 进阶`

- **适用场景**：代码生成 + 审查、多角色讨论、辩论式推理

## 七、多 Agent 系统设计

### 7.1 协作模式

```
┌─────────────────────────────────────────────┐
│              Router / Supervisor             │
│  ┌─────────┐  ┌──────────┐  ┌────────────┐  │
│  │ Agent A  │  │ Agent B  │  │  Agent C   │  │
│  │ (搜索)   │  │ (写作)   │  │  (审查)    │  │
│  └─────────┘  └──────────┘  └────────────┘  │
└─────────────────────────────────────────────┘
```

| 模式 | 描述 | 适用场景 |
|------|------|----------|
| Supervisor | 主 Agent 分发任务给子 Agent | 明确分工的任务 |
| Hierarchical | 多层级管理结构 | 复杂组织协作 |
| Peer-to-Peer | Agent 之间平等对话 | 辩论、头脑风暴 |
| Swarm | 大量简单 Agent 涌现智能 | 大规模并行任务 |

### 7.2 关键设计问题
- **任务分解**：如何将复杂任务拆分给不同 Agent
- **通信协议**：Agent 之间的消息格式与传递方式
- **冲突解决**：多个 Agent 意见不一致时的仲裁
- **共享记忆**：多 Agent 之间如何共享上下文

## 八、生产级 Agent 工程化

### 8.1 评估与监控

- **评估指标**：
  - 任务完成率 / 准确率
  - 工具调用准确率
  - 幻觉率
  - 端到端延迟
- **Tracing 工具**：LangSmith / Langfuse — 追踪每次推理的思考链、Token 消耗、延迟
- **Golden Dataset**：建立标准测试集做自动化回归

### 8.2 部署与交互

- **API 化部署**：封装为 RESTful / WebSocket 服务
- **流式输出**：SSE / WebSocket 实现打字机效果
- **集成渠道**：Slack / 飞书 / Teams / Web UI
- **Human-in-the-loop**：关键步骤需人工确认的机制设计

### 8.3 容错与成本控制

- **容错策略**：超时重试、熔断降级、备用模型路由
- **成本优化**：上下文压缩、缓存策略、小模型分流

- 框架：LangGraph
- LLM：Claude / GPT-4
- 持久化：SQLite / Redis
- 工具：Tavily Search、Python REPL、SQL Query
