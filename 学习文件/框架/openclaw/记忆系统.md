# OpenClaw 记忆系统

## 1. 存储位置总览

所有状态存放在**状态目录**下，默认 `~/.openclaw/`（可通过 `OPENCLAW_STATE_DIR` 覆盖）。

```
~/.openclaw/
├── openclaw.json                              # 主配置文件
├── workspace/                                 # 默认工作区
│   ├── MEMORY.md                              # 主记忆文件（Markdown）
│   ├── memory.md                              # 备选主记忆文件
│   ├── memory/                                # 记忆子目录（所有 *.md 被索引）
│   │   ├── debugging.md
│   │   ├── patterns.md
│   │   └── ...
│   ├── AGENTS.md                              # Agent 指令
│   ├── SOUL.md / IDENTITY.md / USER.md        # 人格/用户信息
│   └── HEARTBEAT.md                           # 心跳提示
├── memory/
│   └── <agentId>.sqlite                       # SQLite 记忆索引（向量 + 全文搜索）
├── agents/
│   └── <agentId>/
│       ├── sessions/
│       │   ├── sessions.json                  # 会话元数据（JSON）
│       │   ├── <sessionId>.jsonl              # 会话转录（JSONL）
│       │   └── <sessionId>-topic-<id>.jsonl   # 话题分支转录
│       └── agent/                             # Agent 内部状态
└── credentials/                               # 凭证
    └── oauth.json
```

相关代码：
- 状态目录解析: `src/config/paths.ts`
- 工作区管理: `src/agents/workspace.ts`
- 记忆文件列举: `src/memory/internal.ts`
- 会话路径: `src/config/sessions/paths.ts`

## 2. 两种记忆：长期记忆 vs 会话记忆

| 维度 | 长期记忆 (Memory) | 会话记忆 (Sessions) |
|---|---|---|
| **格式** | Markdown 文件 (`.md`) | JSONL 文件 (`.jsonl`) |
| **位置** | `~/.openclaw/workspace/MEMORY.md` + `memory/*.md` | `~/.openclaw/agents/<id>/sessions/*.jsonl` |
| **内容** | 用户手写或 Agent 写入的持久知识 | 完整的对话历史（用户+助手+工具调用） |
| **生命周期** | 永久（用户手动管理） | 30 天自动清理，上限 500 条 |
| **索引来源标签** | `"memory"` | `"sessions"` |
| **默认启用** | 是 | 需配置 `experimental.sessionMemory: true` |

### 2.1 长期记忆文件格式

纯 Markdown，用户或 Agent 自由编写：

```markdown
# 项目笔记

## 调试技巧
- 使用 `pnpm test --reporter=verbose` 获取详细输出
- Gateway 日志在 /tmp/openclaw-gateway.log

## 用户偏好
- 偏好 bun 而非 node 运行 TypeScript
- 提交信息用中文
```

### 2.2 会话转录文件格式

JSONL，每行一个 JSON 对象：

```jsonl
{"type":"session","version":1,"id":"abc-123","timestamp":"2026-03-01T10:00:00Z","cwd":"/project"}
{"type":"message","message":{"role":"user","content":"帮我看看 src/app.ts"}}
{"type":"message","message":{"role":"assistant","content":[{"type":"text","text":"让我读取这个文件。"}]}}
{"type":"message","message":{"role":"assistant","content":[{"type":"tool_use","name":"read","input":{"path":"src/app.ts"}}]}}
```

### 2.3 会话元数据 (`sessions.json`)

JSON 字典，key 是会话 key（小写），value 是 `SessionEntry`：

```json
{
  "telegram:user:12345": {
    "sessionId": "abc-123",
    "sessionFile": "/home/user/.openclaw/agents/default/sessions/abc-123.jsonl",
    "channel": "telegram",
    "lastTo": "12345",
    "tokenCount": 15234,
    "model": "claude-sonnet-4-6",
    "updatedAt": "2026-03-01T12:00:00Z"
  }
}
```

写入方式：原子写入（temp 文件 + rename），文件权限 `0o600`。超过 10MB 会触发文件轮转。

相关代码：
- 会话元数据类型: `src/config/sessions/types.ts`
- 会话存储读写: `src/config/sessions/store.ts`
- 会话转录操作: `src/config/sessions/transcript.ts`

## 3. 记忆索引 — SQLite 数据库

位于 `~/.openclaw/memory/<agentId>.sqlite`，是记忆搜索的核心。

### 3.1 数据库 Schema

```sql
-- 索引元信息
CREATE TABLE meta (
  key   TEXT PRIMARY KEY,
  value TEXT NOT NULL          -- JSON: {model, provider, chunkTokens, vectorDims, ...}
);

-- 已索引的文件（md 和 jsonl）
CREATE TABLE files (
  path   TEXT PRIMARY KEY,     -- 相对路径，如 "MEMORY.md" 或 "sessions/abc.jsonl"
  source TEXT NOT NULL,        -- "memory" | "sessions"
  hash   TEXT NOT NULL,        -- 文件内容 SHA-256
  mtime  INTEGER NOT NULL,
  size   INTEGER NOT NULL
);

-- 文本分块 + 向量嵌入
CREATE TABLE chunks (
  id         TEXT PRIMARY KEY,  -- UUID
  path       TEXT NOT NULL,     -- 关联 files.path
  source     TEXT NOT NULL,     -- "memory" | "sessions"
  start_line INTEGER NOT NULL,  -- 在源文件中的起始行
  end_line   INTEGER NOT NULL,
  hash       TEXT NOT NULL,     -- 分块文本 SHA-256
  model      TEXT NOT NULL,     -- 嵌入模型名称
  text       TEXT NOT NULL,     -- 分块原文（~400 tokens / ~1600 字符）
  embedding  TEXT NOT NULL,     -- JSON 浮点数组
  updated_at INTEGER NOT NULL
);

-- 嵌入缓存（避免重复计算）
CREATE TABLE embedding_cache (
  provider     TEXT NOT NULL,
  model        TEXT NOT NULL,
  provider_key TEXT NOT NULL,
  hash         TEXT NOT NULL,
  embedding    TEXT NOT NULL,
  dims         INTEGER,
  updated_at   INTEGER NOT NULL,
  PRIMARY KEY (provider, model, provider_key, hash)
);

-- 全文搜索（FTS5）
CREATE VIRTUAL TABLE chunks_fts USING fts5(text, id UNINDEXED, path UNINDEXED, ...);

-- 向量搜索（sqlite-vec 扩展）
CREATE VIRTUAL TABLE chunks_vec USING vec0(
  id        TEXT PRIMARY KEY,
  embedding FLOAT[<dims>]       -- 维度由首次嵌入决定
);
```

相关代码: `src/memory/memory-schema.ts`

### 3.2 Meta 元信息结构

Meta key 是 `"memory_index_meta_v1"`，value 是 JSON：

```typescript
{
  model: string;           // 嵌入模型名，如 "text-embedding-3-small"
  provider: string;        // 嵌入提供者，如 "openai"
  providerKey?: string;
  sources?: MemorySource[];   // ["memory"] | ["sessions"] | ["memory","sessions"]
  chunkTokens: number;        // 默认 400
  chunkOverlap: number;       // 默认 80
  vectorDims?: number;        // 首次嵌入后设定
}
```

## 4. 索引与搜索机制

### 4.1 分块策略

| 来源 | 分块方式 |
|---|---|
| **Markdown 记忆** | 按 heading 和段落分块，默认 ~400 tokens/块，80 tokens 重叠 |
| **Session JSONL** | 先展平为 `"User: ...\nAssistant: ..."` 格式，再同样分块 |

分块代码: `src/memory/internal.ts` 的 `chunkMarkdown()` 函数（184 行）

### 4.2 搜索方式：混合搜索

```
用户查询 "如何调试 gateway"
    │
    ├─ 向量搜索 (权重 0.7)
    │   查询文本 → embedding → sqlite-vec 近似最近邻搜索
    │
    ├─ 全文搜索 (权重 0.3)
    │   查询关键词 → FTS5 BM25 排名
    │
    └─ 混合排序
        ├─ 可选 MMR（最大边际相关性）去重
        ├─ 可选时间衰减（新内容权重更高）
        └─ 返回 top-K 结果
```

结果格式：

```typescript
{
  path: "MEMORY.md",      // 来源文件
  startLine: 15,           // 起始行
  endLine: 28,             // 结束行
  score: 0.87,             // 相关性分数
  snippet: "...",          // ≤700 字符摘要
  source: "memory"         // "memory" | "sessions"
}
```

### 4.3 嵌入模型支持

| Provider | 模型 |
|---|---|
| OpenAI | `text-embedding-3-small` 等 |
| Google Gemini | `text-embedding-004` 等 |
| Voyage AI | `voyage-3` 等 |
| Mistral | `mistral-embed` 等 |
| Local | `node-llama-cpp`（本地运行，无需 API） |
| auto | 自动选择：优先 OpenAI → 回退本地 |

相关代码:
- 搜索管理器: `src/memory/search-manager.ts`
- 嵌入实现: `src/memory/embeddings-openai.ts`, `embeddings-gemini.ts`, `embeddings-mistral.ts`
- 混合搜索/时间衰减: `src/memory/temporal-decay.ts`

## 5. 索引同步触发时机

| 触发方式 | 场景 | 来源 |
|---|---|---|
| **会话启动时** | 每次 Agent 开始新对话 | `sync.onSessionStart: true` |
| **搜索前** | 如果索引脏了，搜索前先同步 | `sync.onSearch: true` |
| **文件监控** | chokidar 监控 `MEMORY.md` + `memory/**/*.md` | `manager-sync-ops.ts:356` |
| **会话事件** | 转录文件写入后 5 秒去抖触发 | `onSessionTranscriptUpdate` |
| **定时同步** | 可配置间隔（默认关闭） | `sync.intervalMs` |

同步过程：
1. 扫描文件列表，计算 SHA-256 与 mtime
2. 与 `files` 表对比，找出新增/变更/删除的文件
3. 对变更文件重新分块 + 计算嵌入（利用 `embedding_cache` 避免重复计算）
4. **原子写入**：先写到 `.tmp-<uuid>` 临时数据库，再 rename 覆盖（防止索引损坏）

相关代码: `src/memory/manager-sync-ops.ts`

## 6. Agent 记忆工具

Agent 只有两个记忆**读取**工具，没有专用的记忆写入工具：

| 工具 | 功能 | 方向 |
|---|---|---|
| `memory_search` | 语义搜索 MEMORY.md + memory/*.md | **只读** |
| `memory_get` | 按路径+行号读取记忆片段 | **只读** |

相关代码: `src/agents/tools/memory-tool.ts`

系统提示词中指示 Agent 在回答关于先前工作、决策、日期、偏好等问题前，必须先调用 `memory_search`：

## 7. 记忆写入路径 — 不经过 SandboxFsBridge

**记忆的写入和索引完全绕过了 SandboxFsBridge。** 原因是记忆系统在架构上属于 Agent 基础设施，而非 Agent 的工具沙箱。

### 三条独立的写入路径

```
路径 A: Agent 通过 write/edit 工具写 MEMORY.md（间接写入）
路径 B: 会话转录自动追加 JSONL（框架写入）
路径 C: 记忆索引同步到 SQLite（索引系统写入）
```

### 7.1 路径 A：Agent 用通用文件工具写 MEMORY.md

OpenClaw **没有**专用的 "memory_write" 或 "memory_save" 工具。Agent 写入 MEMORY.md 时，用的是**通用的 `write` 或 `edit` 工具**——和它编辑任何其他文件一样。

```
Agent: "我需要记住用户偏好使用 bun"
  │
  ├─ 沙箱关闭时:
  │   edit 工具 → createHostWorkspaceEditTool
  │     → openFileWithinRoot() → node:fs 直接读写
  │     → 写入 ~/.openclaw/workspace/MEMORY.md
  │
  └─ 沙箱开启时:
      edit 工具 → createSandboxedEditTool
        → bridge.readFile() → docker exec cat MEMORY.md
        → bridge.writeFile() → docker exec cat > MEMORY.md
        → 写入的是容器内的挂载路径，但因为 workspace 是 bind mount，
          实际落盘位置仍然是宿主机的 ~/.openclaw/workspace/MEMORY.md
```

关键点：即使在沙箱模式下，workspace 目录通过 Docker bind mount 映射，Agent 在容器内写 MEMORY.md，实际数据仍然写到宿主机磁盘。沙箱只是执行隔离，不是存储隔离。

### 7.2 路径 B：会话转录（JSONL）— 框架直接写入

会话历史由 `@mariozechner/pi-coding-agent` 的 `SessionManager` 自动管理，**完全绕过 SandboxFsBridge**：

```
session.prompt("帮我看看 app.ts")
  │
  ├─ 用户消息 → SessionManager.appendMessage()
  │    → node:fs appendFile() 追加到 ~/.openclaw/agents/<id>/sessions/<sid>.jsonl
  │
  ├─ AI 回复 → SessionManager.appendMessage()
  │    → 同上
  │
  └─ 工具调用/结果 → SessionManager.appendMessage()
       → 同上
```

每条记录是一行 JSON（JSONL 格式），通过 `node:fs/promises.appendFile()` **直接追加到宿主机文件**。路径在 Agent 基础设施层（`~/.openclaw/agents/`），不在 workspace 里，与沙箱完全无关。

### 7.3 路径 C：记忆索引同步（SQLite）— 基础设施层直接写入

```
MEMORY.md 被修改
  │
  ├─ chokidar 检测到文件变更
  │    → markDirty()
  │    → scheduleWatchSync()
  │
  ▼
syncMemoryFiles()
  ├─ node:fs 读取 MEMORY.md 内容
  ├─ 计算 SHA-256，与 files 表对比
  ├─ 如果变更：重新分块 → 调用嵌入 API → 写入 SQLite
  │    → 写入 ~/.openclaw/memory/<agentId>.sqlite
  │    → 使用原子交换：先写 .tmp-<uuid> → rename 覆盖
  └─ 完成
```

### 7.4 为什么记忆不走 SandboxFsBridge

```
┌─────────────────────────────────────────────────────────────────┐
│                      Agent 运行时架构                            │
│                                                                 │
│  ┌──────────────────────┐    ┌──────────────────────┐           │
│  │   工具层（沙箱内）    │    │  基础设施层（宿主机）  │          │
│  │                      │    │                      │           │
│  │  read/write/edit     │    │  SessionManager      │           │
│  │  exec (bash)         │    │  MemoryIndexManager  │           │
│  │  image               │    │  ConfigStore         │           │
│  │                      │    │  GatewayLock         │           │
│  │  ↓ 通过 Bridge ↓    │    │                      │           │
│  │  SandboxFsBridge     │    │  ↓ 直接 node:fs ↓   │           │
│  │  → docker exec       │    │  → ~/.openclaw/      │           │
│  └──────────────────────┘    └──────────────────────┘           │
│           │                           │                         │
│           ▼                           ▼                         │
│  Docker 容器 bind mount          宿主机文件系统                  │
│  (workspace 目录)                 (状态目录)                     │
└─────────────────────────────────────────────────────────────────┘
```

| 组件 | 写入方式 | 原因 |
|---|---|---|
| **Agent 写 MEMORY.md** | 通过 write/edit 工具（沙箱模式走 Bridge，否则直接 node:fs） | MEMORY.md 在 workspace 内，属于"用户文件" |
| **会话转录 JSONL** | 框架直接 `appendFile()`，不走 Bridge | JSONL 在 `~/.openclaw/agents/` 下，是基础设施 |
| **SQLite 索引** | `MemoryManagerSyncOps` 直接操作 SQLite | 索引是基础设施，不受沙箱管控 |
| **sessions.json** | `SessionStore` 直接原子写入 | 元数据是基础设施 |

核心设计原则：**SandboxFsBridge 只管 Agent 工具对 workspace 文件的操作。** 记忆索引、会话存储、配置等基础设施数据由宿主机进程直接管理，不经过沙箱通道。MEMORY.md 比较特殊——它既是"记忆文件"又是"workspace 文件"，所以 Agent 写它时走工具层（可能经过 Bridge），但索引系统读它时走基础设施层（直接 node:fs）。

## 8. 数据流全景

```
Agent 运行中
  │
  ├─ 用户对话 → 写入 sessions/<sessionId>.jsonl (JSONL 追加, node:fs)
  │                  → 更新 sessions/sessions.json (原子替换, node:fs)
  │
  ├─ Agent 写 MEMORY.md → 通过 write/edit 工具 → chokidar 检测变更 → 触发索引同步
  │
  ├─ Agent 调用 memory_search 工具
  │     → 检查索引是否需要同步
  │     → 混合搜索（向量 0.7 + FTS 0.3）
  │     → 返回相关记忆片段给 Agent
  │
  └─ 会话结束 → 会话元数据更新 → 可选触发会话记忆索引

存储层次:
┌─────────────────────────────────────────────────────────────────┐
│ MEMORY.md / memory/*.md        ← 人类可读的持久知识（Markdown） │
├─────────────────────────────────────────────────────────────────┤
│ sessions/*.jsonl               ← 对话历史（JSONL，自动管理）    │
├─────────────────────────────────────────────────────────────────┤
│ sessions.json                  ← 会话路由元数据（JSON，原子写） │
├─────────────────────────────────────────────────────────────────┤
│ memory/<agentId>.sqlite        ← 搜索索引（向量+FTS，原子同步） │
└─────────────────────────────────────────────────────────────────┘
```

**设计哲学**：OpenClaw 的记忆系统是**文件优先**的设计——长期记忆是人类可读的 Markdown，会话历史是可追溯的 JSONL，SQLite 只是搜索索引而非数据源头。索引可以随时从源文件重建，这与传统数据库优先的设计有本质区别。
