  ---
  Agent 沙箱文件桥（SandboxFsBridge）详解

  整体架构

  沙箱文件桥是 Agent 在 Docker
  容器中操作文件的唯一通道。它在宿主机侧做安全校验，然后通过 docker
  exec 在容器内执行实际的文件操作。

  Agent 工具调用
      │
      ▼
  SandboxFsBridge (宿主机 TypeScript)
      │
      ├── 1. 路径解析：filePath → {hostPath, containerPath,
  relativePath}
      ├── 2. 安全校验：openBoundaryFile (宿主机侧)
      ├── 3. 容器内校验：resolveCanonicalContainerPath (容器侧
  readlink -f)
      │
      ▼
  docker exec -i <container> sh -c '<shell script>' -- <args>

  ---
  第一层：路径映射（Mount 系统）

  src/agents/sandbox/fs-paths.ts 中的 buildSandboxFsMounts()
  构建挂载表，有三种来源：

  ┌───────────┬───────────────────────────────────────┬───────────┐
  │   类型    │                 说明                  │  优先级   │
  ├───────────┼───────────────────────────────────────┼───────────┤
  │ workspace │ 主工作区，挂载到容器的 workdir        │ 0（最低） │
  ├───────────┼───────────────────────────────────────┼───────────┤
  │           │ Agent 专属工作区（如果与 workspace    │           │
  │ agent     │ 不同），挂载到                        │ 1         │
  │           │ /openclaw-agent-workspace             │           │
  ├───────────┼───────────────────────────────────────┼───────────┤
  │ bind      │ 用户自定义 bind mount（从配置解析）   │ 2（最高） │
  └───────────┴───────────────────────────────────────┴───────────┘

  路径解析逻辑 (resolveSandboxFsPathWithMounts，fs-paths.ts:98)：
  1. 如果输入是绝对 POSIX 路径（/开头），按容器路径查找匹配的 mount
  2. 否则，按宿主机路径解析后查找匹配的 mount
  3. 找不到匹配的 mount → 抛出 "Path escapes sandbox root" 错误

  mount 列表按路径长度降序排列（最长前缀优先匹配），同长度时 bind >
  agent > workspace。

  ---
  第二层：三重安全校验

  每次文件操作前，assertPathSafety() (fs-bridge.ts:248)
  执行三重检查：

  检查 1 — 词法 Mount 匹配（fs-bridge.ts:249-254）
  const lexicalMount =
  this.resolveMountByContainerPath(target.containerPath);
  if (!lexicalMount) {
    throw new Error(`Sandbox path escapes allowed mounts`);
  }
  确认容器路径在某个已知 mount 下。

  检查 2 — 宿主机边界验证（fs-bridge.ts:256-272）
  const guarded = await openBoundaryFile({
    absolutePath: target.hostPath,
    rootPath: lexicalMount.hostRoot,
    boundaryLabel: "sandbox mount root",
    aliasPolicy: options.aliasPolicy,
  });
  在宿主机侧打开文件，验证：
  - 路径逐段解析，每个符号链接跳转必须留在 root
  内（resolveBoundaryPath）
  - lstat → open(O_NOFOLLOW) → fstat 交叉比对 inode，防止 TOCTOU 竞态
  - 拒绝硬链接（nlink > 1）
  - 返回辨别联合类型 {ok: true, fd} 或 {ok: false, reason}

  检查 3 — 容器内规范路径验证（fs-bridge.ts:274-288）
  const canonicalContainerPath = await
  this.resolveCanonicalContainerPath({...});
  const canonicalMount =
  this.resolveMountByContainerPath(canonicalContainerPath);
  在容器内执行 readlink -f
  获取真实路径（fs-bridge.ts:301-331），再次确认解析后的路径仍在某个
  mount 内。这防止了容器内的符号链接指向 mount 外的区域。

  如果操作是写入，还额外检查 mount 的 writable 标志和全局
  workspaceAccess 配置。

  ---
  第三层：Docker exec 执行

  所有实际 I/O 都通过 runCommand() (fs-bridge.ts:225-246)
  发送到容器内：

  private async runCommand(script: string, options) {
    const dockerArgs = [
      "exec", "-i",
      this.sandbox.containerName,
      "sh", "-c", script,
      "moltbot-sandbox-fs",  // $0，便于 ps 识别
    ];
    if (options.args?.length) {
      dockerArgs.push(...options.args);  // 作为 $1, $2... 传入
    }
    return execDockerRaw(dockerArgs, { input: options.stdin, ... });
  }

  execDockerRaw() (docker.ts:29-107) 用 child_process.spawn("docker",
   args) 执行，支持：
  - stdin 管道传输数据（写文件时用）
  - stdout/stderr Buffer 收集
  - AbortSignal 取消（发送 SIGTERM）
  - allowFailure 标志控制非零退出是否抛异常

  每个文件操作对应的 shell 脚本：

  操作: readFile
  Shell 脚本: set -eu; cat -- "$1"
  说明: 通过 stdout 返回文件内容
  ────────────────────────────────────────
  操作: writeFile
  Shell 脚本: set -eu; dir=$(dirname -- "$1"); if [ "$dir" != "." ];
    then mkdir -p -- "$dir"; fi; cat >"$1"
  说明: 通过 stdin 管道接收数据，自动创建父目录
  ────────────────────────────────────────
  操作: mkdirp
  Shell 脚本: set -eu; mkdir -p -- "$1"
  说明: 递归创建目录
  ────────────────────────────────────────
  操作: remove
  Shell 脚本: set -eu; rm [-f] [-r] -- "$1"
  说明: 根据参数拼接 flags
  ────────────────────────────────────────
  操作: rename
  Shell 脚本: set -eu; dir=$(dirname -- "$2"); if [ "$dir" != "." ];
    then mkdir -p -- "$dir"; fi; mv -- "$1" "$2"
  说明: 自动创建目标父目录
  ────────────────────────────────────────
  操作: stat
  Shell 脚本: set -eu; stat -c "%F|%s|%Y" -- "$1"
  说明: 返回 "类型|大小|mtime"，解析为 SandboxFsStat

  所有脚本使用 set -eu（出错即停、未定义变量报错）和
  --（防参数注入）。参数通过 $1, $2
  位置参数传入而非字符串拼接，避免了命令注入。

  ---
  Docker 容器本身的安全加固

  buildSandboxCreateArgs() (docker.ts:259-368)
  创建容器时施加多重限制：

  - --read-only：只读根文件系统（可选）
  - --cap-drop：丢弃 Linux capabilities
  - --security-opt no-new-privileges：禁止提权
  - seccomp / AppArmor profile 支持
  - --pids-limit、--memory、--cpus、--ulimit：资源限制
  - 环境变量经过 sanitizeEnvVars() 过滤敏感变量
  - validateSandboxSecurity() 在创建前阻止危险的 bind mount、network
  mode 和 profile

  ---
  总结：数据流

  以 writeFile("/app/test.txt", "hello") 为例：

  1. 路径解析：/app/test.txt → 匹配 workspace mount →
  hostPath=/real/workspace/test.txt, containerPath=/app/test.txt
  2. 写权限检查：workspaceAccess === "rw" 且 mount writable === true
  3. 宿主机边界验证：openBoundaryFile 逐段解析
  /real/workspace/test.txt，确认在 workspace root
  内，无符号链接逃逸，无硬链接
  4. 容器内规范路径验证：docker exec sh -c 'readlink -f ...' 确认
  /app/test.txt 解析后仍在 mount 内
  5. 执行写入：docker exec -i <container> sh -c 'set -eu; mkdir -p --
   "$(dirname "$1")"; cat >"$1"' -- /app/test.txt ← stdin 管道送入
  "hello"

  这套设计确保了即使容器内存在恶意符号链接，也无法逃逸到宿主机的非授
  权区域。

⏺ SandboxFsBridge 的用途和使用方式

  一句话概括

  SandboxFsBridge 是 Agent 在 Docker 沙箱容器内操作文件的唯一通道。当
   Agent（AI）需要读写用户工作区的文件时，所有操作都通过这个桥接层在
  容器内执行，而非直接操作宿主机文件系统。

  1. read 工具（src/agents/pi-tools.read.ts:716）

  Agent 读取文件内容时调用：

  readFile: (filePath) => bridge.readFile({ filePath, cwd: root })
  access:   (filePath) => bridge.stat({ filePath, cwd: root })  //
  检查文件是否存在

  场景：Agent 说"让我看看 src/app.ts 的内容"→ 调 read 工具 → docker
  exec cat src/app.ts

  2. write 工具（src/agents/pi-tools.read.ts:734）

  Agent 创建新文件时调用：

  mkdir:     (dir) => bridge.mkdirp({ filePath: dir, cwd: root })
  writeFile: (filePath, content) => bridge.writeFile({ filePath, cwd:
   root, data: content })

  场景：Agent 说"我来创建 src/utils/helper.ts"→ 调 write 工具 →
  docker exec mkdir -p && cat >file

  3. edit 工具（src/agents/pi-tools.read.ts:745）

  Agent 编辑已有文件时调用：

  readFile:  bridge.readFile(...)   // 先读取当前内容
  writeFile: bridge.writeFile(...)  // 写入编辑后的内容
  access:    bridge.stat(...)       // 确认文件存在

  场景：Agent 说"把第 10 行改成 xxx"→ 先读文件 → 应用 diff → 写回

  4. apply_patch 工具（src/agents/apply-patch.ts:228）

  OpenAI 兼容 provider 用的补丁应用工具：

  readFile: bridge.readFile(...)
  writeFile: bridge.writeFile(...)
  remove:   bridge.remove(...)      // 删除文件（如果 patch 要求）
  mkdirp:   bridge.mkdirp(...)      // 新文件需要创建目录
  resolvePath: bridge.resolvePath(...)  // 路径安全校验

  场景：Agent 发出一个 unified diff → apply_patch 逐文件读改写

  5. image 工具（src/agents/tools/image-tool.ts:499）

  Agent 需要查看图片时：

  readFile: (filePath) => bridge.readFile({ filePath, cwd: root })

  场景：Agent 说"让我看看 screenshot.png"→ 从容器内读取二进制图片数据

  6. Prompt

    │     → 容器内校验: docker exec readlink -f /app/src/app.ts →
  仍在 /app/ 下
    │     → 执行: docker exec stat -c "%F|%s|%Y" /app/src/app.ts
    │     → 返回: { type: "file", size: 1234, mtimeMs: ... }
    │
    ├─ bridge.readFile({ filePath: "src/app.ts", cwd: root })
    │     → 同样三重校验
    │     → 执行: docker exec cat /app/src/app.ts
    │     → 返回: Buffer<文件内容>
    │
    ├─ 在 TypeScript 层应用 old→new 替换
    │
    └─ bridge.writeFile({ filePath: "src/app.ts", cwd: root, data:
  修改后内容 })
          → 写权限检查: workspaceAccess === "rw" ✓
          → 三重路径安全校验
          → 执行: echo "修改后内容" | docker exec -i <container> sh
  -c 'cat >"/app/src/app.ts"'

  本质上，SandboxFsBridge 让 Agent 的文件操作看起来像普通 FS
  操作，但实际所有 I/O 都被隔离在 Docker
  容器内，宿主机侧的安全层确保路径不会被滥用。

  第四步：包装中间件（pi-tools.ts:526-538）

  过滤后的工具再逐个包裹三层中间件：

  // 1. Schema 标准化（适配不同 AI provider 的 JSON Schema 要求）
  const normalized = subagentFiltered.map(tool =>
    normalizeToolParameters(tool, { modelProvider })
  );

      agentId,
      sessionKey,

  中间件的执行顺序（洋葱模型，从外到内）：

  AbortSignal 检查
    └→ beforeToolCall Hook
         ├→ 循环检测：连续相同调用超阈值 → 阻断
         ├→ 插件 Hook：before_tool_call → 可阻断或修改参数
         └→ 实际 tool.execute()
              ├→ 参数标准化（normalizeToolParams）
              ├→ 工作区路径守卫（workspaceOnly check）
              └→ 核心执行逻辑（bridge.readFile / fs.readFile / docker exec）

  具体看 wrapToolWithBeforeToolCallHook（pi-tools.before-tool-call.ts:175-233）：

  execute: async (toolCallId, params, signal, onUpdate) => {
    // 1. 运行 Hook 链（循环检测 + 插件 Hook）
    const outcome = await runBeforeToolCallHook({ toolName, params, toolCallId, ctx });
    if (outcome.blocked) {
      throw new Error(outcome.reason);  // 被阻断
    }

    // 2. 执行实际工具逻辑
    try {
      const result = await execute(toolCallId, outcome.params, signal, onUpdate);
      await recordLoopOutcome({ toolName, result });  // 记录结果供循环检测
      return result;
    } catch (err) {
      await recordLoopOutcome({ toolName, error: err });
      throw err;
    }
  }

  配置层级总结

  openclaw.yaml (用户配置)
  ├── tools.allow / tools.deny                    → 全局工具策略
  ├── tools.profile                               → 预设策略模板
  ├── tools.byProvider.<provider>.allow            → 按 AI provider 策略
  ├── tools.exec.*                                → exec/bash 工具配置
  ├── tools.fs.workspaceOnly                      → 文件操作限制在工作区
  ├── tools.loopDetection.*                       → 循环检测配置
  ├── agents.list[].tools.allow                   → 单个 Agent 策略
  ├── sandbox.mode / sandbox.workspaceAccess       → 沙箱开关 + 读写权限
  ├── sandbox.docker.*                            → Docker 容器配置
  └── groups[].tools.allow                        → 群组/频道策略

  核心设计思想：工具的实现（沙箱/宿主机）由运行环境自动决定，工具的可用性由多层策略逐级收紧，工具的执行
  由中间件链保护。Agent 开发者只需声明需要什么工具，不需要关心底层是 Docker exec 还是 node:fs。
